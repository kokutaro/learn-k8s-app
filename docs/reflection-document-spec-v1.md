# 振り返りドキュメント仕様 v1

## 1. 目的

本仕様は `docs/issue-<number>-reflection.md` の書式と内容を統一し、レビュー・運用・将来改修で再利用しやすい記録を残すことを目的とする。

## 2. 対象ドキュメント

- 対象: `docs/issue-<number>-reflection.md`
- 例: `docs/issue-54-reflection.md`
- 1 Issue につき 1 ファイルを原則とする

## 3. 記述ポリシー

- 文体は「である調」に統一する。
- 事実ベースで記述する。推測を含む場合は「推定」と明示する。
- 曖昧語（例: いい感じ、問題ないはず）を避ける。
- 記載根拠がない断定を書かない。
- 情報が欠ける項目は省略せず「未確認」と書く。

## 4. 見出しと章立て

以下を必須章として、順序どおりに記載する。

1. `# Issue #<number> 振り返り`
2. `## 1. 概要`
3. `## 2. 背景と目的`
4. `## 3. 実装内容とポイント`
5. `## 4. レビュー指摘と対応`
6. `## 5. 検証結果`
7. `## 6. 学びと改善アクション`
8. `## 7. 残課題`（任意）
9. `## 8. 参照`（任意）

補足ルール:

- 見出し番号はアラビア数字で統一する（`## 1. ...`）。
- 章の名称は意味を変えない範囲で微調整可。
- 任意章は不要なら省略可。

## 5. 章ごとの必須内容

### 5.1 概要

- 何を解決した Issue か
- 対象範囲（主要機能、画面、コンポーネント、レイヤー）
- 完了判定の要点（例: ブロッカー 0、主要テスト通過）

### 5.2 背景と目的

- 問題が顕在化した背景
- 利用者/運用への影響
- 目的を箇条書きで明記

### 5.3 実装内容とポイント

- 主要変更を「なぜ」「何を」「どう実現したか」で整理
- 変更ファイルまたはモジュールを列挙
- 重要な設計判断（トレードオフ、採用/不採用理由）を記載

### 5.4 レビュー指摘と対応

- 指摘を区分（必須対応 / 任意課題）して記載
- 各指摘に対して対応状況（対応済み / 未対応）を明示
- PR ブロッカーの最終有無を明記

推奨フォーマット（いずれか）:

- 表形式（区分 / 内容 / 判定 / 対応）
- 番号付きリスト（指摘 -> 対応 -> 結果）

### 5.5 検証結果

- 実施した検証種別を列挙（例: lint, typecheck, unit test, integration test, e2e）
- 実行結果を pass/fail で明記
- 失敗が残る場合は、スコープ内外の判定と扱いを記載

### 5.6 学びと改善アクション

- 学び: 今回の実装から得た再利用可能な知見
- 改善アクション: 次回に向けた具体的な行動（2-4件程度）

### 5.7 残課題（任意）

- 本Issueでは未対応だが追跡が必要な課題
- 必要なら起票予定 Issue や backlog 方針を記載

### 5.8 参照（任意）

- 参照した設計書、PR、関連 Issue

## 6. テンプレート

以下を初期テンプレートとして使用する。

```md
# Issue #<number> 振り返り

## 1. 概要
- 対象Issue: #<number>
- 要約: <このIssueで解決したこと>
- 結果: <例: PRブロッカー 0 / 主要検証 pass>

## 2. 背景と目的
- 背景: <問題の発生背景>
- 影響: <ユーザー/運用への影響>
- 目的:
  - <目的1>
  - <目的2>

## 3. 実装内容とポイント
- 変更ファイル/モジュール:
  - <path>
  - <path>
- 主な実装:
  1. <実装内容>
  2. <実装内容>
- 設計判断:
  - <採用した理由>
  - <不採用にした案と理由>

## 4. レビュー指摘と対応
| 区分 | 内容 | 判定 | 対応 |
|---|---|---|---|
| 必須対応 | <指摘> | <対応済み/未対応> | <対応内容> |
| 任意課題 | <指摘> | <対応済み/未対応> | <対応内容または次回対応> |

- 最終判定: PRブロッカー <あり/なし>

## 5. 検証結果
- lint: <pass/fail/未確認>
- typecheck: <pass/fail/未確認>
- test (unit/integration): <pass/fail/未確認>
- e2e: <pass/fail/未確認>
- 補足: <失敗がある場合の理由とスコープ判定>

## 6. 学びと改善アクション
- 学び:
  - <学び1>
  - <学び2>
- 改善アクション:
  1. <次の具体アクション>
  2. <次の具体アクション>

## 7. 残課題
- <未対応課題>

## 8. 参照
- <関連PR/Issue/docs>
```

## 7. 運用ルール

- doc-writer エージェントは本仕様を最優先で参照する。
- 既存の反映文書に揺れがある場合でも、新規作成分は本仕様に合わせる。
- 仕様変更が必要な場合は本ファイルを更新し、必要に応じて doc-writer エージェントの指示も同期更新する。
