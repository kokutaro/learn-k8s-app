# Issue #50 振り返り

## 実装の背景と目的
Issue #50 の主題は、HTTP 409 のエラー表示が常に同じ固定文言になっており、利用者が原因を特定しづらい点の改善でした。

特に、楽観ロック競合（Concurrency）と業務ルール由来の競合（重複割り当てなど）を同一メッセージで扱うと、ユーザーが取るべき次のアクションが不明確になります。

本対応の目的は以下です。
- Concurrency のみを専用文言に限定する
- 非 Concurrency の 409 では原因が分かる文言を返す
- 未知の競合コードでも情報欠落しないフォールバックを用意する

## 実装の内容とポイント
対象はフロントエンド API クライアント層です。

- 変更ファイル
  - osouji-system-frontend/src/lib/api.ts
  - osouji-system-frontend/src/lib/api.test.ts

### 1. 409 競合コードごとの文言マッピングを整理
api.ts に競合コード別の文言マップを定義し、既知コードを個別文言で返すようにしました。

今回の実装では、以下の 3 系統を含む個別文言対応を実施しています。
- 重複メンバー割り当て
- 別エリアへの既割り当て
- 重複スポット

加えて、Concurrency（RepositoryConcurrency）は専用文言を維持し、競合時の再読み込み誘導を明確化しました。

### 2. 409 の未知コード向けフォールバックを追加
既知コードに該当しない場合でも、ユーザーが原因を把握できるように次の順でフォールバックします。
- error.details[0].message
- error.message
- 通信失敗時の汎用文言

これにより、未知コード追加時でも固定文言化を避け、最低限の原因可視化を担保しました。

### 3. 非 409 ステータスの表示方針は維持
400/404/500 などは従来方針を維持し、詳細メッセージ優先の挙動を保持しました。今回の変更は 409 の扱いを主対象に限定しています。

## テスト結果
api.test.ts でエラー表示の期待値を拡充し、以下を検証しました。
- RepositoryConcurrency は専用文言を返す
- 既知の 409 コードは個別文言を返す
- 未知の 409 コードは details.message または message にフォールバックする
- 非 409 とネットワークエラーの既存挙動が維持される

- 全体テスト: 通過

## 今回の実装で学んだこと
- 同じ HTTP ステータスでも、ドメインエラーコードでユーザー体験は大きく改善できる
- Concurrency と業務競合を分離して扱うことが、再試行行動の正確さにつながる
- 未知コードに対するフォールバックは、運用時の事故耐性を高める

## 今後の改善ポイント
- 競合コードと文言定義をドキュメント化し、バックエンドとの契約を明文化する
- 文言管理を i18n テーブルに寄せ、画面間での一貫性を強化する
- 409 以外も含めたエラー表示ガイドラインを整備し、新規 API 追加時の判断を標準化する
